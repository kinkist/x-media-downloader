# x-media-downloader

A Go program that logs into X.com (formerly Twitter) via a real Chrome browser and fetches your home timeline tweets.

## How It Works

1. **First run** — if `cookies.json` does not exist, Chrome opens automatically and navigates to the X.com login page. You log in manually; the program detects the `auth_token` cookie and saves all cookies to `cookies.json`.
2. **Subsequent runs** — `cookies.json` is loaded and used to authenticate the scraper. Your 20 most recent home timeline tweets are printed to stdout.
3. **Cookie auto-save** — after each tweet fetch, the program compares the scraper's cookie jar against the originally loaded values. If X.com updated any cookie during the session (e.g. `ct0` refresh), the changes are written back to `cookies.json` automatically.

## Prerequisites

- Go 1.21+
- Google Chrome installed on your machine

## Getting Started

```bash
# Clone and install dependencies
git clone <repo-url>
cd x-media-downloader
go mod tidy

# Run
go run main.go
```

On the first run, Chrome will open. Log in with your X.com account. The program saves `cookies.json` and exits. Run again to fetch tweets.

To re-login, delete `cookies.json` and run again:

```bash
rm cookies.json && go run main.go
```

## Key Design Decisions

### Why manual login instead of automated login?

X.com's login flow uses [Castle.io](https://castle.io) bot detection. The `LoginEnterUserIdentifierSSO` subtask requires a `castle_token` generated by real browser fingerprinting (mouse movements, Canvas, timing). This token cannot be reproduced in a plain Go HTTP client.

Approaches tried:

| Approach | Result |
|---|---|
| Go HTTP client replicating the login API | Error 399 — `castle_token` required |
| Hardcoding captured `castle_token` | Token expires within minutes |
| chromedp automated login | Castle.io detects headless Chrome |
| **Current: chromedp + manual user login** | **Works reliably** |

### Why poll `auth_token` instead of watching DOM or URL?

X.com redirects to different paths after login depending on account state (2FA, phone verification, etc.). Polling for the `auth_token` cookie is the most reliable signal that authentication succeeded.

### Bearer token for authenticated requests

When `SetCookies` detects an `auth_token` cookie, the scraper switches to `bearerToken1` (the app-level token for authenticated API calls) and sets `isLogged = true`. This ensures GraphQL home timeline requests are sent with the correct credentials.

### Cookie value sanitization

Some X.com cookies (e.g. `twid`, `g_state`, `personalization_id`) store JSON or quoted strings as their value. Go's `net/http` rejects `"` as an invalid byte in cookie values. All `"` characters are stripped via `strings.ReplaceAll` before cookies are passed to the scraper's jar.

### Cookie auto-save after fetch

X.com may refresh short-lived cookies (such as `ct0`, which expires in ~6 hours) during an active API session. After each tweet fetch, `gethometweets` calls `scraper.GetCookies()`, compares each value against the original snapshot (using the already-stripped values to avoid false positives), and writes `cookies.json` only when a change is detected. New cookies added by the server are also appended. All other cookie fields (domain, path, expires, httpOnly, secure) are preserved from the original file.

## Cookie Schema (`cookies.json`)

```json
[
  {
    "name": "auth_token",
    "value": "...",
    "domain": ".x.com",
    "path": "/",
    "expires": 1807228866.0,
    "httpOnly": true,
    "secure": true,
    "sameSite": "None"
  }
]
```

Key cookies:

| Cookie | Purpose |
|---|---|
| `auth_token` | User authentication token (long-lived, ~1 year) |
| `ct0` | CSRF token sent as `X-CSRF-Token` header (short-lived, ~6 hours) |

## Security Notes

- `cookies.json` contains your session token. Add it to `.gitignore` and keep it private.
- File permissions are set to `0600` (owner read/write only) when saved.
